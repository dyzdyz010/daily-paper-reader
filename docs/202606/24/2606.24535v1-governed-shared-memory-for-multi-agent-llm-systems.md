## 摘要
多智能体LLM环境需要稳健的共享知识管理机制。本文形式化了群组记忆问题，并识别了四种基础失效模式：未授权泄露、过时传播、矛盾持续和溯源崩溃。为解决这些问题，我们定义了显式的系统级原语：作用域检索、时序取代、溯源追踪和策略控制的记忆传播。这些原语在MemClaw（一个生产级多租户记忆服务）中实现，并通过ArgusFleet（一个可复现的测试工具，测试四个治理维度）进行评估。与基线比较不同，本研究测量了一个实际生产服务，强调现实世界的架构洞察和负面结果。关键评估结果：溯源方面，成功重建了100%的深度四层推导链，具有正确的写入者身份，且每跳延迟低于一秒；传播方面，展示了高群组内可见性，且零跨群组泄露。在强写入模式下，写入到可见的延迟被优化为单次搜索往返。发现的生产架构问题：作用域执行不对称性——租户隔离得以保持，但子租户作用域最初在直接按ID获取凭据的请求上被绕过，这些凭据是作用域为代理的（在研究过程中披露并修复）；管道排序冲突——虽然矛盾取代对已接纳的写入有效，但同步的近重复门可能会在异步矛盾检测器评估之前过早地拒绝矛盾的写入。结论：仅靠长上下文检索不足以应对生产级多智能体记忆。受控共享记忆需要显式的系统级抽象，而且现场评估对于暴露仅通过设计处理所遗漏的执行和管道排序故障至关重要。

## Abstract
Multi-agent LLM environments require robust mechanisms for shared knowledge management. This paper formalizes the fleet-memory problem and identifies four foundational failure modes: unauthorized leakage, stale propagation, contradiction persistence, and provenance collapse. To address these, we define explicit systems-level primitives: scoped retrieval, temporal supersession, provenance tracking, and policy-governed memory propagation. These primitives are implemented in MemClaw, a production multi-tenant memory service, and evaluated via ArgusFleet, a reproducible harness testing four governance dimensions. Rather than a baseline comparison, this study measures a live production service, emphasizing real-world architectural insights and negative results. Key Evaluation Results Provenance: Successfully reconstructed 100% of depth-four derivation chains with correct writer identity at sub-second per-hop latency. Propagation: Demonstrated high intra-fleet visibility with zero cross-fleet leakage. Under strong write mode, write-to-visible latency was optimized to a single search round-trip. Production Architectural Issues Discovered Asymmetric Scope Enforcement: Tenant isolation held, but sub-tenant scope was initially bypassed on direct GET-by-id requests for agent-scoped credentials (disclosed and remediated during the study). Pipeline Ordering Conflict: While contradiction supersession works for admitted writes, a synchronous near-duplicate gate can prematurely reject contradictory writes before the asynchronous contradiction detector can evaluate them. Conclusion: Long-context retrieval alone is insufficient for production multi-agent memory. Governed shared memory demands explicit systems-level abstractions, and live evaluation is vital to expose enforcement and pipeline-ordering failures missed by design-only treatments.

- **核心问题定义**：将多智能体LLM的共享记忆形式化为 **受控分布式状态问题**，不再是检索质量问题，而是分布式系统的访问控制、时序一致性、溯源、同步与策略传播问题。
- **四大失败模式**：论文明确识别了直接危害系统正确性的四种缺陷——  
  - 未授权泄露（越权读取）  
  - 过时传播（更新未能同步到全部代理）  
  - 矛盾持久化（冲突记忆共存无解）  
  - 溯源崩溃（无法追踪记忆来源）
- **整体意图**：提出并验证一套系统架构与治理原语，把AI记忆从“上下文窗口问题”提升为“受控分布式记忆问题”，并在真实生产服务中暴露设计与部署阶段的薄弱环节。

## 二、方法论：核心思想与关键技术设计

### 2.1 群组记忆的形式化模型
- 系统定义为 `F = (A, M, G, P, T)`：  
  - `A`：代理集合  
  - `M`：共享记忆基板  
  - `G`：治理与策略层  
  - `P`：溯源元数据  
  - `T`：时序排序与取代语义  
- 写操作定义为 `w_i = (a_i, c_i, s_i, t_i, p_i)`，其中 `s_i` 为可见作用域，写是一种状态转移。
- 检索操作定义为 `r_j = f(q_j, a_j, g_j, t_j)`，引入治理约束 `g_j` 与时序正确性条件 `t_j`。
- 提出不变量 **Inv‑Scope**：任意代理 `a_j` 仅能检索到其有权访问的作用域 `s_i` 内的记忆，确保作用域执行无旁路。

### 2.2 架构原语
- **作用域划分**：代理本地、团队共享、租户全局、受限策略四层。
- **时序解析**：记忆携带时间戳、取代引用、矛盾标记等，通过时序与策略驱动的取代规则解决冲突。
- **溯源图**：每条记忆关联写入者身份、来源系统、派生历史与修改谱系。
- **策略控制检索**：流程为：语义候选生成 → 策略过滤 → 时序解析 → 溯源增强 → 排序交付。与传统纯相似度检索有本质区别。

### 2.3 实现与评估工具
- **MemClaw**：生产级多租户记忆服务，暴露 REST API（写入、搜索、ID直取、实体更新、重分配），身份通过 `mc_` 前缀密钥解析，注入 `X-Agent-ID`，租户边界强制，子租户作用域最初有缺口。
- **ArgusFleet**：开源可复现评估套件（Python 3.12），包含四个实验分别针对四种失败模式，支持异步 HTTP 客户端、结构化事件日志、确定性种子与 JSONL 追溯。

## 三、实验设计

- **实验场景**：所有测试均针对一个实时生产服务 `memclaw.net` 的单独租户，无基线对比，强调测量生产服务本身。
- **四组实验及其设计**：
  - **泄露实验**：4 舰队 × 6 代理，写入舰队及代理作用域的秘密，通过语义搜索与 ID 直取共 192 次探测（期望拒绝 164、期望允许 28），测量“泄露率”与“缺失率”。
  - **矛盾实验**：100 个事实（如 `billing-service-N` 的 `deployment_region`），使用顺序与并发写入相互矛盾的 RDF 三元组，共 400 次写入，测量矛盾检测率、过时读取率及写延迟。
  - **溯源实验**：50 条深度为 4 的派生链（共 200 次写入），通过 `derived_from` 元数据回追，衡量完整性、写入者身份准确性、深度保真度及每跳延迟。
  - **传播实验**：分速率阶段（40 个舰队作用域事实，探测舰队同胞与非舰队的可见性及泄露）和窗口阶段（8 个专用事实，以 250 ms 轮询测量写入到可见的真实延迟）。
- **对比方法**：无传统基线对比；报告的“负结果”（作用域漏洞、管道冲突）自身构成对照证据，说明单纯设计无法发现的执行缺陷。

## 四、资源与算力

- 论文中 **未提及任何 GPU 型号、数量，也无模型训练环节**。评估完全基于 REST API 客户端请求，算力开销集中在测试脚本执行与网络 I/O，不涉及高密度计算。此举也印证其定位：测量一个分布式状态服务，而非模型训练或推理性能。

- **总体规模**：泄露 192 探针，矛盾 400 写入（200 事实对），溯源 200 写入（50 条链），传播 40 事实 + 200 探针，窗口阶段 8 事实高粒度轮询。
- **重复性与确定性**：种子与计划可复现，每条结果有 JSONL 事件追踪，四个独立矛盾运行验证稳定率。
- **充分性**：样本量对揭露作用域缺口、管道竞争及即时可见性等定性架构问题已足够；但针对统计精度的泄漏率、跨舰队泄漏上限等，作者承认需要更大规模（如数量级提升）才能收紧置信区间。
- **客观性**：实验中刻意报告不利结果（自身服务的作用域旁路、去重门抢先问题），并公开修复后复测追踪，减少了选择性报道偏差。

## 六、主要结论与发现

- **溯源原语稳健**：深度四推导链 100% 重建，写入者身份准确无误，每跳中位延迟 291 ms，亚秒级即可追溯完整谱系。
- **传播正确且低延迟**：舰队同胞可见率 0.975，零跨舰队泄露。强写入模式下写入到可见窗口仅一次搜索往返（中位 0.83 s），一致性代价由写入时同步富集吸收，而非读时滞后。
- **矛盾取代条件成立**：当两次冲突写入都被接纳时，取代机制完全有效（90/90，100% 正确）；但整体检测率仅 0.49，原因在于**同步近重复去重门**在异步矛盾检测器运行之前就拒绝了大量语义相似的冲突写入。
- **作用域执行的不对称性**：租户级隔离在所有路径生效，但子租户（舰队/代理）作用域在直接 `GET /memories/{id}` 上未被实施，即使网关解析了代理身份也同样被忽略。该漏洞在研究期间被披露并修复，修复后复测零泄露。
- **核心论据成立**：仅靠长上下文检索无法满足多智能体生产需求；必须引入系统级受控共享记忆抽象，且真实环境的现场评估能暴露设计阶段无法看到的强制执行与管道排序错误。

## 七、优点

- **问题形式化清晰**：将 AI 记忆提升为分布式系统问题，由失败模式驱动架构设计，概念严谨。
- **生产级实现与开放评估**：提供可运行的记忆服务及可复现的 ArgusFleet，允许状态测度从 JSONL 确定性重建。
- **负结果的诚实披露**：主动报告自身服务的漏洞和管道缺陷，并展示“预测→测量→修复”的完整循环，增强可信度。
- **分离一次性效应与真实服务器行为**：通过专用窗口阶段剥离客户端调度偏差，准确测量写入到可见的真实延迟，避免测量伪影。
- **治理与溯源原生整合**：同时解决作用域、时间、溯源与传播，超越单纯检索增强，对安全敏感场景有直接价值。

## 八、不足与局限

- **自评估偏差**：评估针对自研服务，缺乏独立审计，尽管通过负结果和公开追踪部分缓解，仍存在潜在利益冲突。
- **无对比基线**：未与传统长上下文方案或其他记忆系统比较，难以量化治理原语带来的相对提升。
- **实验规模有限**：单租户、小样本（192/200 级别），跨租户隔离等关键主张未经实证；代理作用域探针虽经聚焦测试，但覆盖面较窄。
- **传播窗口测量限定**：时间到可见度基于 8 个事实、250 ms 粒度及低负载，生产高并发下尾部延迟可能恶化；仅测强写入模式，其他模式未覆盖。
- **管道冲突修复处于建议阶段**：去重门与矛盾检测的排序问题尚未在生产中重新设计，仅作分析，未给出修复后复测。
- **未测量最终一致性长尾**：矛盾取代异步约 6 s 内完成，但若负载持续增加，取代延迟的稳定性未被测试。
