## 摘要
LLM智能体越来越成为个人助理，必须记住用户几个月的个人资料：他们是谁（属性），他们日常做什么（习惯），他们偏好什么（偏好），并随着工作、日常和品味的变化而更新。现有基准通过简短、简化的交互评估这种“记忆”能力，但缺少真实行为的三个核心特性：个人资料是异质的，属性、习惯和偏好在不同时间线上演变；变化由外部环境驱动，如季节和生活事件；证据很少明确陈述，而是分散在不同应用中的许多小动作，记忆系统必须从中推断。我们引入DynamicMem，一个合成基准，为每个用户构建15个月的活动，提供长期多应用数据，而真实用户的隐私使其无法获取。它提供用户一致的轨迹，平均每个用户有220万token和1772个有据可查的事件，跨16个应用如电子商务、健身和社交平台。个人资料在此期间演变，从未明确给出：每个属性、习惯或偏好必须从分散在应用中的小信号推断。我们在五个季度检查点评估，以跟踪系统随着历史增长如何扩展。对五个代表性系统进行基准测试揭露了单一准确率隐藏的问题：(i) 个人资料重建随历史长度下降，而服务任务准确率保持平稳，尽管两者都使用相同的内存；(ii) 没有系统既能保持仍然正确的事实，又能替换改变的事实，错误集中在偏好和命名确切指称上；(iii) 超过93%的失败可追溯到内存检索的内容，而不是写答案的模型，因此最大的改进空间在于内存本身。代码：https://wenyaxie023.github.io/DynamicMem/

## Abstract
LLM agents increasingly act as personal assistants that must remember a user's profile over months: who they are (attributes), what they routinely do (habits), and what they prefer (preferences), and keep it updated as jobs, routines, and tastes drift. Existing benchmarks evaluate this "memory" ability through short, simplified interactions, missing three core properties of real behavior: the profile is heterogeneous, with attributes, habits, and preferences evolving on different timelines; changes are driven by external context such as seasons and life events; and evidence is rarely stated explicitly, instead scattered across many small actions in different apps that a memory system must infer from. We introduce DynamicMem, a synthetic benchmark that constructs 15 months of activity per user, providing long-term multi-app data that real users' privacy keeps out of reach. It provides user-consistent trajectories averaging 2.2M tokens and 1,772 grounded events per user across 16 applications such as e-commerce, fitness, and social platforms. The profile evolves over this period and is never given explicitly: each attribute, habit, or preference must be inferred from small signals scattered across apps. We evaluate at five quarterly checkpoints to track how systems scale as history grows. Benchmarking five representative systems exposes problems a single accuracy score hides: (i) profile reconstruction degrades with history length while service-task accuracy stays flat, despite both drawing on the same memory; (ii) no system both keeps facts that stay true and replaces facts that change, with errors clustering on preferences and on naming the exact referent; and (iii) over 93% of failures trace to what the memory retrieves, not to the model writing the answer, so the largest room for improvement lies in memory itself. ### 2. 论文提出的方法论

`DynamicMem`的核心思想是通过一个可扩展的合成数据管道，生成具有内在一致性的、长达15个月的跨应用用户行为轨迹。其关键技术细节分为三个阶段：

*   **阶段一：多时间尺度用户档案构建 (`User Modeling with Multi-Timescale Dynamics`)**
    *   **架构分解**：将用户档案分解为**静态基础档案**（人口统计、社会经济因素等）和**动态档案**。动态档案在K=6个生活领域（如工作与教育、健康与自我关怀）中演变，并进一步分解为三种具有不同变化机制的类别：
        *   **属性 (Attributes)**：离散事实或拥有物，随生活事件突变（如获得新工作）。
        *   **习惯 (Habits)**：常规行为模式，适应中期环境（如季节性晨跑）。
        *   **偏好 (Preferences)**：长期倾向，缓慢漂移（如从快餐转向健康饮食）。
    *   **因果演化机制**：用户状态的变化并非随机，而是遵循**世界背景 → 类型化增量 → 状态更新**的因果链。
        *   **世界背景**：为每个季度窗口生成个性化的世界背景，包含季节、重大事件和外部语境。
        *   **类型化增量**：根据世界背景，对属性/习惯/偏好进行受限的操作（如`ADD`/`MODIFY`、`ACQUIRE`/`ADJUST`、`SHIFT`/`REFINE`），并要求每个操作都有明确的因果理由。一个世界事件常驱动跨类别的耦合变化。

*   **阶段二：意图条件化事件链生成 (`Intent-Conditioned Event Chain Generation`)**
    *   为了提供可靠的多事件证据，而非噪声单点事件，此阶段生成**事件链**。
    *   每个事件链是由同一用户意图驱动的、在单个领域和窗口内的一系列有序事件，用于展示一个状态变化（如图3所示，一个“获取专业认证”的意图驱动了在Gmail、LinkedIn和WhatsApp上的一系列事件）。

*   **阶段三：状态一致的多应用日志生成 (`State-Consistent Multi-app Log Generation`)**
    *   将事件链中的每个事件“落地”为16个有状态应用（如Amazon、Spotify、Gmail）中的一个具体API调用日志。
    *   **有状态的应用环境**是关键：每个应用维护自身的状态（如购物车、播放列表、收件箱），每次API调用都会读取并确定性地更新此状态，确保了在应用内和应用间实体引用的一致性和行为合理性，避免了幻觉 ID。

*   **评估协议**
    *   **检查点评估**：在5个季度检查点上评估记忆系统，而不是在整个轨迹末尾。这可以分析性能如何随历史长度缩放，以及系统能否正确覆盖已变化的信息。
    *   **任务家族**：在每个检查点，使用两类任务进行评估：
        1.  **状态完成 (State Completion)**：恢复用户当前的属性、习惯或偏好（例如，填写一个JSON模式的空白字段）。
        2.  **个性化服务 (Personalized Service)**：基于恢复的状态提供个性化辅助（例如，根据用户的习惯生成提醒，或根据偏好预配置过滤器）。

### 6. 论文的主要结论与发现

*   **发现一：两种任务性能随历史长度出现分化。** 随着历史增长（从C1到C5），**状态完成**（重建用户档案）准确率持续下降，而**个性化服务**（基于档案执行任务）准确率保持稳定甚至略有提升。这是因为状态完成更依赖检索单一、确切的“金标准”日志，而个性化服务可以从轨迹中越来越丰富的相关上下文中受益。
*   **发现二：状态完成的衰退主要由隐式的偏好驱动。** 在属性、习惯和偏好三类档案中，偏好项的准确率下降最为显著。这归因于偏好很少被直接陈述，必须从大量零散行为中推断，而历史越长，无关日志越多，信号被稀释。
*   **发现三：记忆系统存在“保留”与“更新”两种独立的失败模式，且无设计能两全。**
    *   例如，A-Mem在习惯的长期保留上表现最佳，因为其追加式笔记机制会不断重申事实。而MemoryOS的层级化压缩因批次分割问题，习惯保留表现不稳定。
    *   在状态更新方面，SimpleMem的递归整合会“淹没”只出现一次但始终有效的偏好陈述，HippoRAG2和RAG的追加式存储则能更好地处理状态变更。
*   **发现四：在个性化服务中，习惯类任务表现极差，瓶颈在于“主动例程选择”。** 所有系统在状态完成中能成功回忆的习惯，在个性化服务（如根据周日早上的咖啡场景生成提醒）中却大幅失败。原因是模型无法在没有明确提示的情况下，将当前场景与记忆中的多个习惯进行匹配，以选出正确的例程。
*   **核心瓶颈定位（错误分析）：** **超过93%的失败可追溯到记忆系统检索/提供的证据不足或不当，而非最后的答案生成模型。** 最大的改进空间在于记忆存储和检索机制本身。
