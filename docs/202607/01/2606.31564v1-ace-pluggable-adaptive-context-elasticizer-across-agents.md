## 摘要
代理任务的复杂性日益增加，导致轨迹长度迅速增长，这对基于固定上下文窗口的大语言模型（LLM）代理提出了重大挑战。现有的上下文管理技术，如截断和摘要，存在固有的不灵活性和不可逆性：一旦信息被丢弃或压缩，即使在后续决策步骤中变得至关重要，也无法恢复。为了解决这些限制，我们提出了自适应上下文弹性器（ACE），这是一个即插即用的模块，可以在每个决策步骤弹性地将历史步骤信息编排到代理的上下文中。ACE维护一个无损的消息维护层，为每个历史步骤存储原始消息和压缩的抽象，而上下文编排层则在每个决策步骤根据当前任务状态自适应地为每个步骤分配弹性类型，即原始、抽象或丢弃。这种可逆的设计确保主LLM始终接收到一个紧凑但信息丰富的上下文。我们将ACE适配到四个不同的代理框架，包括ReAct、DeepAgent、WebThinker和MiroFlow，无需训练或架构修改。实验表明，ACE始终优于截断和摘要基线，并在所有四个代理框架上带来了一致的性能提升。

## Abstract
The increasing complexity of agentic tasks has led to rapidly growing trajectory lengths, which poses significant challenges for large language model (LLM) based agents with fixed context windows. Existing context management techniques, such as truncation and summarization, suffer from inherent inflexibility and irreversibility: once information is discarded or compressed, it cannot be recovered even when it becomes critically relevant in later decision steps. To address these limitations, we propose the Adaptive Context Elasticizer (ACE), a plug-and-play module that elastically orchestrates historical step information into the agent's context at each decision step. ACE maintains a lossless message maintenance layer that stores both raw messages and compressed abstractions for each historical step, while a context orchestration layer adaptively assigns each step an elastic type as raw, abstract, or drop, at every decision step based on the current task state. This reversible design ensures that the main LLM always receives a compact yet information-rich context. We adapt ACE to four diverse agent frameworks, including ReAct, DeepAgent, WebThinker, and MiroFlow, without training or architectural modifications. Experiments show that ACE consistently outperforms truncation and summarization baselines, and brings consistent performance gains across all four agent frameworks.

### 2. 方法论：自适应上下文弹性器（ACE）

- **核心思想**：将“实际决策上下文”的构建与“原始历史消息”解耦。ACE作为外部包装器，不修改原有智能体框架，而是在每个决策步动态编排历史信息。

- **两层架构设计**：
    - **消息维护层 (Message Maintenance Layer)**：**无损**存储每个历史步骤的完整信息。它对每个步骤 `i` 保存两部分内容：
        - **原始消息** (`h_raw_i`)：步骤产生的完整推理、动作和观察。
        - **压缩抽象** (`h_abs_i`)：由辅助LLM生成的摘要，仅保留关键事实（如数字、URL、日期）和显式的后续任务信号。该层信息**永不修改**，保证了完整性。
    - **上下文编排层 (Context Orchestration Layer)**：在每个决策步 `t`，根据当前任务状态，为历史步骤 `i` 动态分配一个“弹性类型” `d_t^i`，决定其在该步上下文中的呈现形式：
        - **原始 (Raw)**：直接使用 `h_raw_i`。
        - **抽象 (Abs)**：使用压缩后的 `h_abs_i`。
        - **丢弃 (Drop)**：不纳入当前上下文。
        最终的上下文由三部分组成：**固定的任务描述 + 经弹性编排的历史步骤(i=1到t-2) + 强制保持为原始形式的最新步骤(t-1)**。

- **关键流程与算法**：
    1. **决策生成**：弹性器（本身也是一个LLM）以任务描述和所有历史步骤的**压缩抽象**作为输入，预测当前决策步 `t` 下每个历史步骤 `i` 的弹性类型 `d_t^i`。
    2. **上下文构建**：根据 `{d_t^i}` 和固定规则，从消息维护层提取信息，组装成当步的上下文 `C_t`。

        T, // 任务描述

- **机理优势**：此设计是**可逆**和**灵活**的。一个被压缩或丢弃的步骤，在后续任何需要它的决策步中，可以随时恢复为原始信息或重新纳入上下文，实现了真正随任务状态演进而变化的“弹性”上下文。

### 8. 不足与局限
- **算力与成本**：虽然无需训练，但每个步骤都需要向LLM额外请求生成压缩摘要和编排决策，这**增加了推理时的API调用成本和响应延迟**，可能对实时性要求高的场景不够友好。
- **对弹性器LLM的依赖**：弹性编排的性能高度依赖于弹性器LLM本身的理解和决策能力。如果该LLM能力较弱，可能会做出错误的编排决策（如丢弃关键信息），从而适得其反。当前实验中，弹性器与主LLM能力相同，在较弱模型上的效果未知。
- **摘要质量的风险**：采用固定LLM（GPT-4o）生成摘要，其质量直接影响后续编排。如果初始摘要未能捕获全部关键细节，且该信息在后续步骤才变得关键，那么即使恢复为“摘要”形式，也可能因信息丢失而导致决策失误。
- **应用场景限制**：该方法的收益主要在长轨迹、复杂任务中体现。对于短平快的任务，其额外开销可能超过其带来的收益。
